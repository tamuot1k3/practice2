## Вопрос 1

Что выведет этот код? Объясните порядок выполнения блоков.

```java
class Demo {
    static { System.out.print("A "); }
    { System.out.print("B "); }
    Demo() { System.out.print("C "); }
    public static void main(String[] args) {
        System.out.print("D ");
        new Demo();
        System.out.print("E ");
        new Demo();
    }
}
```

**Предсказанный вывод:**
```
A D B C E B C
```

**Объяснение:**
Сначала JVM загружает класс и выполняет статический блок - выводится A. Потом запускается main и печатает D. При первом new Demo() срабатывает блок экземпляра B, а за ним конструктор C. Дальше main печатает E, и при втором new Demo() опять блок экземпляра B и конструктор C. Статический блок запускается только один раз при загрузке класса, а блок экземпляра - каждый раз перед конструктором.

## Вопрос 2

Класс `Child` в пакете `b`, класс `Parent` в пакете `a`. `Child extends Parent`. Какие члены `Parent` доступны в `Child`? А какие доступны в другом классе пакета `b`, который НЕ наследует `Parent`?

**Ваш ответ:**
В Child доступны public и protected члены. Public - потому что видны всем, protected - потому что Child является наследником Parent, даже если они в разных пакетах. Package-private и private члены недоступны - первые видны только внутри пакета a, вторые только внутри самого класса Parent.
Если же в пакете b есть другой класс, который не наследует Parent, то ему доступны только public члены. Protected уже не подходит, потому что этот класс не наследник.

## Вопрос 3

Можно ли создать sealed-класс, у которого все наследники — record? Какое преимущество это даёт при использовании в switch?

**Ваш ответ:**
Да, можно. Record по умолчанию является final, а это как раз одно из требований sealed (наследник должен быть final, sealed или non-sealed).
Главное преимущество - компилятор знает все допустимые типы, поэтому switch с паттерн-матчингом можно писать без default. Если потом добавить новый тип в permits, компилятор сам покажет все switch-и, где этот тип не обработан. Плюс record позволяет удобно доставать поля прямо в case.

## Вопрос 4

Что выведет этот код? Объясните, что такое String Pool и как работает `intern()`.

```java
String a = "Hello";
String b = new String("Hello");
String c = b.intern();
System.out.println(a == b);
System.out.println(a == c);
System.out.println(b == c);
```

**Предсказанный вывод:**

```
false
true
false
```

**Объяснение:**
String Pool - это специальная область памяти, где JVM хранит строковые литералы. Когда мы пишем "Hello", строка попадает в пул. Переменная a ссылается на этот объект в пуле. new String("Hello") создает отдельный объект в куче, поэтому b указывает на другой объект. Метод intern() ищет строку с таким содержимым в пуле и возвращает ссылку на неё - а там уже лежит та же строка, на которую указывает a. Поэтому a == c дает true, а a == b и b == c - false, потому что b это отдельный объект в куче.

## Вопрос 6

В чём разница между `StringBuilder` и `StringBuffer`? Приведите конкретный сценарий, когда нужен именно `StringBuffer`.

**Ваш ответ:**
StringBuilder не потокобезопасен и работает быстрее. StringBuffer потокобезопасен - его методы синхронизированы, но из-за этого он медленнее.
Пример, когда нужен StringBuffer: несколько потоков параллельно пишут логи в один общий буфер. Если использовать StringBuilder, данные могут перемешаться или потеряться.

## Вопрос 7

Что выведет этот код? Объясните, что происходит с переменной `x` в лямбде.

```java
int x = 10;
Runnable r = () -> System.out.println(x);
// x = 20;  // что будет, если раскомментировать?
r.run();
```

**Предсказанный вывод:**

```
10
```

**Объяснение:**
Лямбда захватывает переменную x. В Java лямбды могут использовать только effectively final переменные - те, которые не меняются после присвоения. Здесь x не меняется, поэтому все работает.
Если раскомментировать x = 20, код вообще не скомпилируется - переменная перестанет быть effectively final, и компилятор выдаст ошибку.

## Вопрос 8

Напишите пример лямбда-выражения, которое нельзя заменить на ссылку на метод. Объясните почему.

**Ваш пример и объяснение:**


```java
Function<String, String> exclaim = s -> s + "!";
```
Здесь выполняется конкатенация, а не вызов какого-то одного метода. Ссылка на метод может указывать только на конкретный существующий метод, а операция + таковым не является.
Еще пример: s -> s.length() > 5 - тут сначала вызывается метод, а потом идет сравнение. Это составное выражение, и его нельзя выразить одной ссылкой на метод.

## Вопрос 9

Чем `EnumMap` быстрее `HashMap` для ключей-enum? Опишите внутреннюю реализацию в одном предложении.

**Ваш ответ:**

EnumMap хранит значения в обычном массиве, где индекс - это ordinal() enum-константы, поэтому доступ идет за O(1) без хеширования и без коллизий, в отличие от HashMap с его бакетами и хеш-функциями.

## Вопрос 10

Дан `record Point(int x, int y) {}`. Что сгенерирует компилятор автоматически? Можно ли добавить в record изменяемое поле? Можно ли наследоваться от record?

**Ваш ответ:**

Компилятор сам создаст: конструктор Point(int x, int y), геттеры x() и y(), методы equals(), hashCode() и toString(). Все поля будут private final.
Добавить изменяемое поле нельзя - все компоненты record финальные. Можно добавить только статические поля.
Наследоваться от record тоже нельзя - он неявно final. Сам record тоже не может наследовать другой класс, потому что уже наследует java.lang.Record. Зато может реализовывать интерфейсы.
